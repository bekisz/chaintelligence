# PRD: Multi-Chain Coin Table Redesign

## 1. Problem Statement

The current `coin` table uses `symbol VARCHAR(8)` as the primary key and stores a single `ethereum_address` column. This design has several critical limitations:

1. **No multi-chain support**: A token like USDC exists on Ethereum, Arbitrum, Base, BSC, and Solana with different contract addresses — but only one address can be stored.
2. **Dummy addresses**: Tokens without an Ethereum contract receive placeholder values like `0xdummy_sol`, making the `ethereum_address` column unreliable for on-chain lookups.
3. **Hardcoded token address maps**: Each network (Arbitrum, Base, BNB) maintains a separate hardcoded dictionary in `uniswap_utils.py` (`ARBITRUM_TOKEN_ADDRESSES`, `BASE_TOKEN_ADDRESSES`, `BNB_TOKEN_ADDRESSES`) — these drift out of sync with the DB.
4. **Symbol collisions**: `VARCHAR(8)` PK means two different tokens sharing the same symbol cannot coexist (e.g., bridged `USDC.e` vs native `USDC` on Arbitrum).
5. **Slow lookups by `coin_id`**: `coin_id` was added retroactively as a `SERIAL` column without being promoted to PK; JOINs from the `swaps` table go through a non-PK index.
6. **No CoinMarketCap `contract_address` per-platform extraction**: The ingestion DAG (`coin_ingestion.py`) currently only extracts the Ethereum contract address from CMC's `/v2/cryptocurrency/info` response, ignoring all other platforms.

### Scope

This PRD covers:
- Redesign of the `coin` table to use `coin_id` as PK
- Introduction of a new `coin_contract` table for multi-chain contract addresses
- Updates to the coin ingestion DAG to populate multi-chain contracts from CMC
- A config-driven list of supported chains
- Migration strategy for all FK-dependent tables

---

## 2. Goals & Non-Goals

### Goals
- **G1**: `coin_id SMALLINT` becomes the true, immutable primary key of the `coin` table — all other tables FK to it.
- **G2**: A new `coin_contract` table stores one row per (coin, chain) combination with the verified contract address.
- **G3**: The coin ingestion DAG reads the CMC `/v2/cryptocurrency/info` `contract_address` array and stores entries for all configured chains.
- **G4**: Supported chains are driven by `config/chains.yaml`; default: `ethereum`, `arbitrum`, `base`, `bsc`, `solana`.
- **G5**: Lookups by `coin_id` are O(1) — the column is `SMALLINT PRIMARY KEY` (B-tree index by default).
- **G6**: The hardcoded `ARBITRUM_TOKEN_ADDRESSES`, `BASE_TOKEN_ADDRESSES`, `BNB_TOKEN_ADDRESSES` dicts are replaced with a single DB-backed loader.
- **G7**: Backward compatibility: `symbol` remains `UNIQUE NOT NULL` so existing queries/views using symbol continue working.

### Non-Goals
- Redesigning the `swaps` or `liquidity_pool` table structure beyond FK changes.
- Adding new chains beyond the initial five.
- Decentralized exchange contract verification (e.g., Etherscan verified source).

## 4. Proposed Design

### 4.1 New Schema

#### `coin` Table (Modified)

```sql
CREATE TABLE coin (
    coin_id SMALLINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    symbol VARCHAR(10) NOT NULL,               -- widened from 8 to 10
    name VARCHAR(255),
    slug VARCHAR(255),
    hardness INTEGER DEFAULT 0,
    cmc_rank INTEGER,
    cmc_id INTEGER UNIQUE,                     -- CMC's own ID, made UNIQUE
    image_url TEXT,
    price NUMERIC,
    price_timestamp TIMESTAMPTZ,
    decimals INTEGER DEFAULT 18,               -- default decimals (Ethereum)
    percent_change_1h NUMERIC,
    percent_change_24h NUMERIC,
    percent_change_7d NUMERIC,
    percent_change_30d NUMERIC,
    percent_change_60d NUMERIC,
    percent_change_90d NUMERIC,
    market_cap NUMERIC,
    market_cap_dominance NUMERIC,
    fully_diluted_market_cap NUMERIC,
    tvl NUMERIC,
    total_supply NUMERIC,
    circulating_supply NUMERIC,
    max_supply NUMERIC,
    first_historical_data TIMESTAMPTZ,
    cmc_last_updated TIMESTAMPTZ,
    CONSTRAINT uq_coin_symbol UNIQUE (symbol)
);

-- Fast lookup indexes
CREATE INDEX idx_coin_cmc_id ON coin (cmc_id);
CREATE INDEX idx_coin_symbol_upper ON coin (UPPER(symbol));
```

> **Key changes**:
> - `coin_id SMALLINT GENERATED BY DEFAULT AS IDENTITY` is now the **PK** (max 32,767 — more than enough for tracked coins).
> - `symbol` remains `UNIQUE` for backward compat but is **not** the PK.
> - `symbol` widened to `VARCHAR(10)` to accommodate tokens like `USDC.E`, `USDT0`.
> - `ethereum_address` column **removed** — replaced by `coin_contract` table.
> - `cmc_id` made `UNIQUE` to allow deterministic upsert by CMC ID.

#### `coin_contract` Table (New)

```sql
CREATE TABLE coin_contract (
    coin_id SMALLINT NOT NULL REFERENCES coin(coin_id) ON DELETE CASCADE,
    chain VARCHAR(20) NOT NULL,                -- e.g. 'ethereum', 'arbitrum', 'base', 'bsc', 'solana'
    contract_address VARCHAR(64) NOT NULL,     -- 42 for EVM, 44 for Solana base58
    decimals INTEGER DEFAULT 18,
    is_native BOOLEAN DEFAULT FALSE,           -- TRUE for ETH on ethereum, SOL on solana, etc.
    verified_at TIMESTAMPTZ,                   -- when this address was last confirmed
    PRIMARY KEY (coin_id, chain)
);

-- Fast lookup by contract address per chain
CREATE UNIQUE INDEX idx_coin_contract_addr ON coin_contract (chain, LOWER(contract_address));

-- Fast lookup by chain alone
CREATE INDEX idx_coin_contract_chain ON coin_contract (chain);
```

> **Design rationale**:
> - Composite PK `(coin_id, chain)` ensures exactly one address per coin per chain.
> - `contract_address VARCHAR(64)` accommodates both EVM (42 chars) and Solana base58 (44 chars).
> - The unique index on `(chain, LOWER(contract_address))` prevents duplicate addresses on the same chain.
> - `decimals` is per-chain since tokens can have different decimals on different chains (e.g., USDC is 6 on Ethereum but potentially different on other chains).

### 4.3 Updated Coin Ingestion DAG

#### Task: `cmc_info_fetch` (Modified)

Extract contracts for **all configured chains**, not just Ethereum:

```python
# Load chain config
chains_config = yaml.safe_load(open('config/chains.yaml'))
platform_to_chain = {
    c['cmc_platform_name']: c['name'] 
    for c in chains_config['chains']
}

# In the API response loop:
contracts = info.get('contract_address', [])
chain_addresses = {}
if isinstance(contracts, list):
    for c in contracts:
        platform_name = c.get('platform', {}).get('name', '')
        chain = platform_to_chain.get(platform_name)
        if chain:
            chain_addresses[chain] = {
                'address': c.get('contract_address'),
                'decimals': c.get('decimals')  # if available
            }

detailed_info[cid]['chain_addresses'] = chain_addresses
```

#### Task: `cmc_upsert_to_db` (Modified)

After upserting into `coin`, also upsert into `coin_contract`:

```python
# Upsert coin_contract for each chain
for chain, addr_info in info.get('chain_addresses', {}).items():
    cur.execute("""
        INSERT INTO coin_contract (coin_id, chain, contract_address, decimals, verified_at)
        VALUES (
            (SELECT coin_id FROM coin WHERE symbol = %s),
            %s, %s, %s, NOW()
        )
        ON CONFLICT (coin_id, chain) DO UPDATE SET
            contract_address = EXCLUDED.contract_address,
            decimals = COALESCE(EXCLUDED.decimals, coin_contract.decimals),
            verified_at = NOW()
    """, (symbol, chain, addr_info['address'].lower(), addr_info.get('decimals', 18)))
```

### 4.4 DB-Backed Token Address Loader

Replace the hardcoded dicts in `uniswap_utils.py` with a single function:

```python
def load_token_addresses_for_chain(chain: str) -> Dict[str, str]:
    """Load symbol → contract_address mapping for a given chain from DB."""
    mapping = {}
    conn = psycopg2.connect(DATA_WAREHOUSE_DB)
    cur = conn.cursor()
    cur.execute("""
        SELECT UPPER(c.symbol), cc.contract_address 
        FROM coin_contract cc
        JOIN coin c ON cc.coin_id = c.coin_id
        WHERE cc.chain = %s
    """, (chain.lower(),))
    for row in cur.fetchall():
        mapping[row[0]] = row[1]
    cur.close()
    conn.close()
    return mapping
```

This replaces `ARBITRUM_TOKEN_ADDRESSES`, `BASE_TOKEN_ADDRESSES`, `BNB_TOKEN_ADDRESSES`, and the Ethereum token loading in `config.py`.

## 5. Performance Considerations

### 5.1 Why `SMALLINT` PK?

| Type | Size | Max Value | JOIN cost |
|------|------|-----------|-----------|
| `SERIAL (INT)` | 4 bytes | 2.1B | Baseline |
| `SMALLINT` | 2 bytes | 32,767 | ~50% less I/O per row |
| `VARCHAR(8)` (current PK) | 9–10 bytes | — | 4–5× more I/O per row |

With ~500 tracked coins and millions of swap rows, switching from `VARCHAR(8)` JOINs to `SMALLINT` JOINs reduces index size significantly and improves cache hit rates.

### 5.2 Index Strategy

| Index | Purpose |
|-------|---------|
| `coin(coin_id)` PK | O(1) lookup for all FK JOINs |
| `coin(symbol)` UNIQUE | Backward-compat queries by symbol |
| `coin(cmc_id)` UNIQUE | Deterministic CMC upsert |
| `coin_contract(coin_id, chain)` PK | Per-chain address lookup |
| `coin_contract(chain, LOWER(contract_address))` UNIQUE | Reverse lookup: address → coin |

## 6. Migration Strategy

### Phase 1: Schema Evolution (Non-Breaking)
1. Run migration SQL to:
   - Make `coin_id` the PK (requires dropping old PK, adding new)
   - Add `UNIQUE` constraint on `symbol`
   - Widen `symbol` to `VARCHAR(10)`
   - Create `coin_contract` table
2. Add `coin0_id` / `coin1_id` to `liquidity_pool` (nullable initially)
3. Add `coin_id` to `coin_family` and `coin_price_history` (nullable initially)

### Phase 2: Data Backfill
1. Run backfill scripts to populate:
   - `coin_contract` from existing `ethereum_address` values (chain = 'ethereum')
   - `coin0_id` / `coin1_id` in `liquidity_pool` from symbol JOINs
   - `coin_id` in `coin_family` and `coin_price_history`
2. Trigger updated coin ingestion DAG to fetch CMC contracts for all configured chains

### Phase 3: Code Migration
1. Update `coin_ingestion.py` to populate `coin_contract` for all chains
2. Replace hardcoded `ARBITRUM_TOKEN_ADDRESSES`, `BASE_TOKEN_ADDRESSES`, `BNB_TOKEN_ADDRESSES` with `load_token_addresses_for_chain()`
3. Update `postgres_fetcher.py`, `route_analyzer.py`, and `api/main.py` to use `coin_id` JOINs where currently using symbol strings
4. Update `config.py` → `load_tokens_from_db()` to use `coin_contract` table

### Phase 4: Cleanup
1. Drop old `ethereum_address` column from `coin`
2. Drop old `coin0_symbol` / `coin1_symbol` from `liquidity_pool` (or keep as denormalized cache)
3. Drop hardcoded token address dicts from `uniswap_utils.py`
4. Remove `0xdummy_*` placeholder addresses
